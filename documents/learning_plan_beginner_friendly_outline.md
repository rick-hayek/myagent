## 1. 学习路线总览（新手视角）

本大纲保持现有阶段编号与目录结构不变，目标是把“知道学什么”升级为“知道做到什么算完成”。

- Phase 1：打通 LLM 基础交互链路（API -> Prompt -> Parser/Structured Output -> Streaming）
- Phase 2：打通工具调用闭环（Tool Schema -> Tool Call -> Execution Loop -> 自动代理执行）
- Phase 3：打通短期/长期记忆机制，学会管理上下文成本
- Phase 4：打通 RAG 检索与回答质量优化
- Phase 5：打通复杂任务规划与执行控制（ReAct/LangGraph）
- Phase 6：打通多智能体协作与职责拆分
- Phase 7：打通系统级 Agent（代码修改、命令执行、安全边界）
- Phase 8：打通评估与追踪，建立可量化迭代能力

学习节奏建议：
- 每个 Phase 按“必做 -> 验收 -> 总结 -> 补选做”推进
- 每完成 1 个 Phase，至少沉淀 1 份“可复用模板”到文档

## 2. 每阶段统一模板

每个 Phase 都按以下模板记录，避免“做了很多但无法证明已掌握”。

### 2.1 目标
- 本阶段要解决的核心问题是什么
- 完成后能独立实现什么能力

### 2.2 前置知识
- 必须提前掌握的概念、依赖或运行环境

### 2.3 必做任务（MUST）
- 本阶段必须完成的最小任务集合（建议 3-6 项）

### 2.4 选做任务（NICE TO HAVE）
- 强化理解或提升工程质量的扩展任务

### 2.5 验收标准（可量化）
- 至少 3 条可观测、可复现、可判断通过/失败的标准

### 2.6 常见坑与排错
- 本阶段最常见 3-5 个问题
- 每个问题对应定位思路和修复动作

### 2.7 输出物（代码/文档/截图）
- 至少包含：可运行代码、阶段总结文档、关键运行结果截图或日志

## 3. Phase 1-8 新手增强版目录大纲

### Phase 1：基础理论与大模型交互（`01_basic_api`）
- 目标：
  - 通过纯 SDK 和 LangChain 两条链路调用模型，理解统一消息对象
  - 掌握 Prompt 模板、输出解析、结构化输出、流式输出
- 前置知识：
  - Python 基础、虚拟环境、环境变量
- 必做任务（MUST）：
  - 跑通 `1_ai_sdk_call.py` 与 `2_langchain_basic.py`
  - 跑通 `3_prompt_template.py`、`4_output_parser.py`、`5_streaming.py`
  - 增加最小错误处理：Key 缺失、请求异常、解析失败
- 选做任务（NICE TO HAVE）：
  - 同一任务对比 `JsonOutputParser` 与 `with_structured_output` 失败率
  - 增加简单 CLI 参数输入（替代硬编码）
- 验收标准（可量化）：
  - 5 个脚本均可在本机运行并产生预期输出
  - 至少演示 1 次 API 失败并输出可读错误信息
  - 至少演示 1 次结构化输出成功返回并可访问字段
- 常见坑与排错：
  - 模型名或依赖版本不一致导致调用失败
  - 误把 `AIMessage` 当字符串处理
  - `.env` 未加载或 Key 命名错误
- 输出物：
  - 运行日志截图、Phase 1 总结、错误处理说明

### Phase 2：工具调用与外部交互（`02_calculator_agent` + `03_weather_agent`）
- 目标：
  - 理解工具调用协议与执行循环
  - 能区分模型“下达调用指令”和程序“真实执行工具”的职责
- 前置知识：
  - Phase 1 完成；函数、类型注解、JSON Schema 基础
- 必做任务（MUST）：
  - 跑通 `1_basic_tools.py`、`2_bind_tools_to_llm.py`
  - 跑通 `3_agent_execution_loop.py`，理解为何单次 `if tool_calls` 不够
  - 跑通 `4_langgraph_agent.py`，对比自动循环与手动循环
  - 增加 guardrails：`max_steps`、未知工具兜底、工具异常回传
- 选做任务（NICE TO HAVE）：
  - 演示并行工具调用与有依赖的顺序调用
  - 在 `03_weather_agent` 接入真实 API 并统一错误处理
- 验收标准（可量化）：
  - 至少 3 个问题可稳定触发正确工具调用
  - 至少 1 个问题演示多轮工具调用链路
  - 至少 1 个未知工具或工具报错场景可被优雅处理
- 常见坑与排错：
  - 忘记回传 `ToolMessage` 或 `tool_call_id`
  - 无步数限制导致循环不可控
  - 工具参数类型不匹配导致运行时异常
- 输出物：
  - 工具调用日志（含 tool name/args/result）、Phase 2 总结文档

### Phase 3：上下文与记忆机制（`04_memory_chatbot_v1`、`05_memory_chatbot_v2`）
- 目标：
  - 让 Agent 支持多轮对话并具备可控记忆策略
- 前置知识：
  - Phase 1-2；会管理消息历史

  - 实现短期记忆与长期记忆两版聊天
  - 为历史消息设置长度上限或摘要策略

## 4. 通用工程基线（从 Phase 1 开始）

### 4.1 环境与依赖一致性检查清单
- 锁定 Python 版本（建议 3.10+）
- 每个 Phase 标注最小依赖集与安装命令
- 统一 `.env.example` 字段说明，避免文档与代码不一致

### 4.2 错误处理模板（API / Schema / Tool）
- API 错误：网络异常、限流、鉴权失败 -> 明确错误码与重试策略
- Schema 错误：结构化输出校验失败 -> 降级重试或提示用户修正输入
- Tool 错误：参数错误、运行时异常 -> 捕获后回传可解释错误

### 4.3 日志最小规范
- 必须记录：`request_id`、`phase`、`tool_name`、`latency_ms`、`token_usage`
- 关键节点都要有日志：模型请求、工具执行、结果回传、异常分支

### 4.4 安全最小规范
- 输入默认不可信，禁止直接拼接到高风险命令
- 工具使用白名单，默认只读优先
- 文件系统操作限制在指定工作目录
- 循环执行必须有 `max_steps` 和人工中断点

### 4.5 测试最小规范
- 每个能力至少 1 个 happy path
- 每个能力至少 1 个 failure path
- 每个 Phase 至少 1 个回归用例（防止后续改坏）

## 5. 阶段完成判定规则（Definition of Done）

每个 Phase 完成必须同时满足：

- 功能跑通证据：
  - 可执行命令、输入样例、输出结果齐全
- 验收用例：
  - 至少 3 个通过用例，覆盖核心能力
- 失败场景演示：
  - 至少 1 个失败场景，且有明确报错与处理策略
- 文档同步：
  - `README` 或教程更新，记录“做了什么、为什么、如何复现”
- 复盘沉淀：
  - 至少 3 条阶段经验（踩坑、权衡、下阶段改进点）
